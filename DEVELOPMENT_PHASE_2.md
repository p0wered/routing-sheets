## 1. Обзор изменений

### Суть переработки

Система переходит от ручного создания планов, маршрутных листов и операций к **автоматическому формированию** на основе справочной информации:

| Было (Фаза 1) | Стало (Фаза 2) |
|---|---|
| Планы создаются вручную через UI | Планы предзаполнены в БД, только просмотр и закрытие |
| МЛ создаются вручную | МЛ формируются автоматически из плана |
| Операции добавляются вручную | Операции формируются автоматически из состава изделия |
| Начальник цеха — один на всю систему | Начальник цеха — один на каждый цех |
| Все данные видны всем | Данные фильтруются по цеху пользователя |
| Разбиение МЛ переносит выбранные операции в новый МЛ | Разбиение МЛ делит количество (МЛ и его операции) |
| Нет разбиения операций | Операцию можно разбить по количеству |
| Детали и состав изделий отсутствуют | Детали и состав изделий заданы в БД (только чтение) |

### Новый бизнес-процесс

```
1. Планы производства предзаполнены в БД
   (Цех №358, январь: изделие №1 × 20 шт., изделие №2 × 10 шт.)
                    │
2. Начальник/Плановый отдел выбирает месяц и план
                    │
3. Нажимает «Сформировать МЛ» для конкретного плана
                    │
4. Система автоматически:
   a) Берёт изделие из плана и его состав (детали × кол-во)
   b) Для каждой детали берёт все привязанные операции (1 деталь → N операций)
   c) Умножает кол-во деталей на кол-во изделий в плане
   d) Создаёт МЛ со всеми операциями (статус: Черновик)
                    │
5. При необходимости — разбиение:
   - МЛ разбивается по количеству (10 → 7 + 3)
   - Операции в разбитых МЛ делятся пропорционально
   - Отдельную операцию тоже можно разбить по количеству
                    │
6. Управление статусами:
   - План: Открыт → Закрыт
   - МЛ: Черновик → Активен → Завершен/Отменен
   - Операции: Ожидает → В работе → Завершена/Отменена
```

### Пример автоформирования

**Исходные данные:**
- Изделие «ГП110» состоит из: Крышка ×1, Плата ×1, Болт ×2
- Для крышки две операции: «Штамповка заготовки» (Штамповочный, 30₽) и «Шлифовка крышки» (Механический, 20₽)
- Для платы одна операция: «Сборка платы» (Сборочный, 200₽)
- Для болта одна операция: «Токарная обработка болта» (Механический, 10₽)

**План:** Произвести 3 шт. ГП110

**Сформированный МЛ (Quantity = 3):**

| № | Операция | Деталь | Кол-во | Цена | Сумма |
|---|----------|--------|--------|------|-------|
| 1 | Штамповка заготовки | Крышка | 3 (1×3) | 30₽ | 90₽ |
| 2 | Шлифовка крышки | Крышка | 3 (1×3) | 20₽ | 60₽ |
| 3 | Сборка платы | Плата | 3 (1×3) | 200₽ | 600₽ |
| 4 | Токарная обработка болта | Болт | 6 (2×3) | 10₽ | 60₽ |

## 2. Новая модель данных

### 2.1. Part (Деталь)

Деталь — элемент, из которого состоит изделие. Данные заданы заранее в БД (только чтение через API).

| Поле | Тип | Описание |
|------|-----|----------|
| Id | int | PK |
| Name | string(300) | Название детали (например, «Крышка», «Плата», «Болт М6») |
| Description | string(1000)? | Описание (опционально) |

**Связи:**
- Part → ProductParts (1:N)
- Part → PartOperations (1:N)

### 2.2. PartOperation (Операция для изготовления детали)

Описывает операцию, которую нужно выполнить для изготовления конкретной детали. У одной детали может быть несколько операций. Данные заданы заранее в БД (только чтение через API).

| Поле | Тип | Описание |
|------|-----|----------|
| Id | int | PK |
| PartId | int | FK → Part |
| SeqNumber | int | Порядковый номер операции в рамках детали |
| Name | string(300) | Название операции (например, «Штамповка заготовки») |
| Code | string(50)? | Код операции (опционально) |
| OperationTypeId | int? | FK → OperationType (тип операции) |
| GuildId | int? | FK → Guild (цех, в котором выполняется операция) |
| Price | decimal(18,2)? | Цена за единицу |

**Связи:**
- PartOperation → Part (N:1)
- PartOperation → OperationType (N:1)
- PartOperation → Guild (N:1)

### 2.3. ProductPart (Состав изделия)

Связующая таблица: какие детали входят в какое изделие и в каком количестве. Данные заданы заранее в БД (только чтение через API).

| Поле | Тип | Описание |
|------|-----|----------|
| Id | int | PK |
| ProductItemId | int | FK → ProductItem |
| PartId | int | FK → Part |
| Quantity | int | Кол-во деталей на одно изделие |

**Связи:**
- ProductPart → ProductItem (N:1)
- ProductPart → Part (N:1)
- Уникальный индекс: (ProductItemId, PartId)

### 2.4. PlanStatus (Статус плана производства)

| Поле | Тип | Описание |
|------|-----|----------|
| Id | int | PK |
| Code | string(50) | Код статуса |
| Name | string(100) | Название |

**Предзаполненные значения:**

| ID | Code | Name |
|----|------|------|
| 1 | OPEN | Открыт |
| 2 | CLOSED | Закрыт |

### Диаграмма новых связей

```
Guild ─────────────┬── PlanPosition (+ GuildId, PlanMonth, PlanYear, StatusId)
                   │        │
                   │        └── ProductItem
                   │               │
 User (+ GuildId)  │         ProductPart (кол-во деталей на изделие)
                   │               │
                   │             Part (деталь)
                   │               │
                   │         PartOperation (1:N операций на деталь)
                   │               │
                   ├── PartOperation.GuildId (цех изготовления)
                   │
                   └── RoutingSheet ── Operation (автоформирование)
```

## 3. Изменения в существующих моделях

### 3.1. User — добавление привязки к цеху

| Поле | Изменение | Описание |
|------|-----------|----------|
| GuildId | **ДОБАВИТЬ** (int?) | FK → Guild. Обязателен для WorkshopChief и WorkshopForeman. NULL для PlanningDept |

**Ограничения:**
- **WorkshopChief**: один на цех — уникальный индекс (GuildId) с фильтром WHERE Role = 'WorkshopChief'. При создании проверять, что в этом цехе ещё нет начальника

### 5.3. Автоформирование МЛ

| Метод | Путь | Описание | Авторизация |
|-------|------|----------|-------------|
| POST | `/api/RoutingSheets/generate/{planPositionId}` | Сформировать МЛ для плана | WorkshopChief, PlanningDept |

**Логика эндпоинта генерации:**

```
POST /api/RoutingSheets/generate/{planPositionId}

1. Найти PlanPosition по ID
2. Проверить, что план в статусе OPEN
3. Проверить, что для этого плана ещё нет МЛ
4. Проверить доступ по цеху (для WorkshopChief — только свой цех)
5. Получить ProductItem → его ProductParts (детали + кол-во)
6. Для каждого ProductPart:
   a) Получить все PartOperations этой детали
   b) Для каждой PartOperation создать Operation:
      - Name = PartOperation.Name
      - Code = PartOperation.Code
      - OperationTypeId = PartOperation.OperationTypeId
      - GuildId = PartOperation.GuildId
      - Price = PartOperation.Price
      - Quantity = ProductPart.Quantity × PlanPosition.QuantityPlanned
      - Sum = Price × Quantity
      - StatusId = 1 (Ожидает)
      - SeqNumber = порядковый номер (сквозная нумерация)
7. Создать RoutingSheet:
   - Number = автогенерация (МЛ-YYYY-NNNN)
   - Name = "МЛ для {ProductItem.Name}"
   - PlanPositionId = planPositionId
   - ProductItemId = PlanPosition.ProductItemId
   - UnitId = 1 (шт.)
   - StatusId = 1 (Черновик)
   - Quantity = PlanPosition.QuantityPlanned
   - Operations = [сформированные операции]
8. Сохранить в БД
9. Вернуть созданный МЛ с операциями
```

### 5.4. Разбиение МЛ (по количеству)

| Метод | Путь | Описание | Авторизация |
|-------|------|----------|-------------|
| POST | `/api/RoutingSheets/{id}/split` | Разбить МЛ по количеству | WorkshopChief, PlanningDept |

**Тело запроса:**

```json
{ "splitQuantity": 3 }
```

**Логика:**

```
POST /api/RoutingSheets/{id}/split  { splitQuantity: 3 }

1. Найти RoutingSheet по ID (с операциями)
2. Валидация:
   - splitQuantity > 0
   - splitQuantity < routingSheet.Quantity
   - Для каждой операции: (operation.Quantity × splitQuantity) должно делиться
     на routingSheet.Quantity нацело (чтобы получились целые числа)
3. Вычислить remainingQuantity = routingSheet.Quantity - splitQuantity
4. Создать новый RoutingSheet:
   - Number = автогенерация
   - Name = копия из оригинала
   - PlanPositionId, ProductItemId, UnitId = копия
   - StatusId = 1 (Черновик)
   - Quantity = splitQuantity
5. Для каждой операции оригинала:
   - newOpQuantity = operation.Quantity × splitQuantity / originalQuantity
   - remainingOpQuantity = operation.Quantity - newOpQuantity
   - Создать копию операции в новом МЛ с Quantity = newOpQuantity
   - Обновить оригинальную операцию: Quantity = remainingOpQuantity
   - Пересчитать Sum = Price × Quantity для обеих
6. Обновить оригинальный МЛ: Quantity = remainingQuantity
7. Сохранить всё в БД
8. Вернуть оба МЛ
```

### 5.5. Разбиение операции (по количеству)

| Метод | Путь | Описание | Авторизация |
|-------|------|----------|-------------|
| POST | `/api/Operations/{id}/split` | Разбить операцию по количеству | Все роли |

**Тело запроса:**

```json
{ "splitQuantity": 4 }
```

**Логика:**

```
POST /api/Operations/{id}/split  { splitQuantity: 4 }

1. Найти Operation по ID
2. Валидация:
   - splitQuantity > 0
   - splitQuantity < operation.Quantity
3. Создать новую операцию в том же МЛ:
   - Скопировать все поля (Name, Code, OperationType, Guild, Price, Status)
   - Quantity = splitQuantity
   - Sum = Price × splitQuantity
   - SeqNumber = следующий после максимального в этом МЛ
   - PerformerId = NULL (исполнитель не копируется)
4. Обновить оригинальную операцию:
   - Quantity = operation.Quantity - splitQuantity
   - Sum = Price × новый Quantity
5. Сохранить в БД
6. Вернуть обе операции
```
