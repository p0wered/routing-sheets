# RoutingSheets — админ-панель маршрутных листов

Веб-приложение для предприятия: планирование производства, **автоматическое формирование маршрутных листов (МЛ)** по составу изделия и технологическим операциям деталей, учёт операций по цехам, назначение исполнителей, смена статусов, разбиение МЛ и операций **по количеству**, управление пользователями и справочниками.

Система состоит из **ASP.NET Core 8 Web API** (SQLite, EF Core, JWT) и **React SPA** (Vite, TypeScript, Tailwind CSS 4).

## Возможности

- **Авторизация:** JWT (Bearer), пароли в БД через BCrypt; claim `GuildId` для фильтрации данных по цеху.
- **Планы производства:** позиции плана привязаны к **цеху**, **месяцу и году**, изделию и плановому количеству; статусы **Открыт / Закрыт**; просмотр и смена статуса (не ручное создание планов через API в текущей версии).
- **Маршрутные листы:** формирование **одной кнопкой** из позиции плана (`POST .../generate/{planPositionId}`) на основе состава изделия (`ProductPart`) и операций деталей (`PartOperation`): для каждой детали изделия создаётся **отдельный маршрутный лист** со своими операциями; смена статуса МЛ; **разбиение МЛ по количеству** с пропорциональным пересчётом операций.
- **Операции:** просмотр в составе МЛ и **по цеху**; назначение/снятие исполнителя; смена статуса; **разбиение операции по количеству** в рамках того же МЛ.
- **Справочники с CRUD:** единицы измерения, цеха, исполнители (доступ в UI для начальника цеха и планового отдела — см. меню).
- **Пользователи:** CRUD только для роли **Начальник цеха**; обязательный выбор цеха для начальника и мастера; один начальник на цех (проверка на бэкенде).
- **Чтение без CRUD в UI/API (по замыслу):** детали (`Part`), связь деталь–операции, состав изделия (`ProductPart`), изделия, статусы МЛ/операций/планов — задаются данными и миграциями/seed.
- **UX:** уведомления (Sonner), сортировка таблиц, **локализация ru/en** (i18next), печать/выгрузка отчёта по МЛ (pdfmake).

Подробные матрицы прав, история «фазы 1» и целевая «фаза 2» задокументированы в корневых файлах [ПЛАН_РАЗРАБОТКИ.md](DEVELOPMENT_PHASE_1.md) и [ПЛАН_РАЗРАБОТКИ_ФАЗА_2.md](DEVELOPMENT_PHASE_2.md).

## Архитектура

```
Браузер
  └── React SPA (Vite) — маршрутизация, AuthContext, TanStack Query, axios → /api/*
         │
         │  HTTPS (в dev: Vite proxy /api → backend)
         ▼
  ASP.NET Core Web API
  ├── JWT Bearer + политики [Authorize] / [Authorize(Roles=...)]
  ├── Контроллеры (CRUD и бизнес-операции)
  └── EF Core → SQLite (`backend/Data/routing.db`)
```

При старте приложения выполняется `Database.Migrate()` и при **пустой** таблице пользователей — начальное заполнение тестовых учётных записей.

## Настройка

### Строка подключения к БД

Файл: `backend/appsettings.json`, секция `ConnectionStrings:DefaultConnection`.

По умолчанию: `Data Source=Data/routing.db` — при старте API путь разрешается относительно каталога проекта `backend/`, файл создаётся в `backend/Data/routing.db`. Чтобы использовать другой файл, укажите относительный путь от `backend/` или абсолютный `Data Source=/полный/путь/к файлу.db`.

### JWT

Секция `Jwt` в `appsettings.json`: `Key`, `Issuer`, `Audience`, `ExpireMinutes` (по умолчанию 480 минут).

Для **продакшена** обязательно замените `Key` на длинный секрет из переменных окружения или секрет-хранилища; не коммитьте реальные секреты.

### CORS

В `backend/Program.cs` разрешены источники `http://localhost:5173` и `https://localhost:5173` (Vite). При другом origin добавьте его в политику `AllowFrontend`.

## Запуск для разработки

### 1. База данных

Ничего настраивать не нужно: при первом запуске API файл SQLite создаётся автоматически, схема применяется через `Database.Migrate()`.

### 2. Backend

```bash
cd backend
dotnet run
```

В `backend/Properties/launchSettings.json` два профиля:

- **`http` (по умолчанию при `dotnet run`):** только `http://localhost:5145` — с ним совпадает прокси Vite в `vite.config.ts`.  
- **`https`:** `https://localhost:7050` и тот же HTTP-порт — запуск: `dotnet run --launch-profile https`; тогда в Vite укажите `target: 'https://localhost:7050'` и `secure: false`.

Swagger: `/swagger` на том порту, на котором слушает API.

Если порт занят, измените `applicationUrl` в `launchSettings.json` и `vite.config.ts`.

Доверие к dev-сертификату (для профиля **https**):

```bash
dotnet dev-certs https --trust
```

### 3. Frontend

```bash
cd frontend
npm install
npm run dev
```

Приложение: `http://localhost:5173`.

В `frontend/vite.config.ts` запросы `/api` проксируются на **`http://localhost:5145`** (профиль `http`). Если переключили API на другой URL/порт, обновите `server.proxy['/api'].target`.

### Порядок

1. `dotnet run` в `backend/`  
2. `npm run dev` в `frontend/`  
3. Браузер → `http://localhost:5173`

## Учётные записи и роли

Начальные пользователи создаются в `backend/Program.cs` **только если** таблица `Users` пуста:

| Логин    | Пароль      | Роль              | Цех (GuildId) |
|----------|-------------|-------------------|---------------|
| `chief1` | `chief123`  | Начальник цеха    | 1 (первый цех в seed) |
| `chief2` | `chief123`  | Начальник цеха    | 2 |
| `foreman1` | `foreman123` | Мастер цеха    | 1 |
| `planner` | `planner123` | Плановый отдел | нет (все цеха) |

Константы ролей в коде: `WorkshopChief`, `WorkshopForeman`, `PlanningDept`.

**Кратко по доступу в UI**

- **Маршрутные листы** (`/`) и **Операции по цехам** (`/operations`) — все три роли; данные начальника/мастера фильтруются по цеху пользователя; плановый отдел может выбирать цех там, где предусмотрено.
- **Справочники** (`/references`) — начальник цеха и плановый отдел.
- **Пользователи** (`/users`) — только начальник цеха.

Детальная матрица (включая различия «кто может формировать МЛ, закрывать план, менять статусы») приведена в [ПЛАН_РАЗРАБОТКИ_ФАЗА_2.md](DEVELOPMENT_PHASE_2.md).

## Предметная область и бизнес-логика

- **Позиция плана** — заказ/строка плана на период (цех + месяц/год + изделие + количество + статус открыт/закрыт).
- **Изделие** — номенклатурная позиция; **состав** задаётся через `ProductPart` (деталь × количество на одно изделие).
- **Деталь** — узел; для детали в справочнике заданы **операции изготовления** (`PartOperation`: тип, цех, цена, порядок).
- **Формирование МЛ:** для выбранной открытой позиции плана без уже созданных МЛ API обходит состав изделия. Для **каждой детали** из `ProductPart` создаётся **отдельный маршрутный лист**, а внутри него — операции из `PartOperation` этой детали. Количество МЛ и его операций рассчитывается как  
  `количество_детали_в_составе × плановое_количество_изделий`; новый МЛ создаётся в статусе «Черновик».
- **Разбиение МЛ:** задаётся количество, отделяемое в новый МЛ; доли операций пересчитываются пропорционально; проверка целочисленности операций при делении.
- **Разбиение операции:** часть количества переносится в новую операцию в том же МЛ; исполнитель у новой операции не копируется.

Статусы (идентификаторы в БД совпадают с использованием в коде):

- **МЛ:** 1 Черновик → 2 Активен → 3 Завершён / 4 Отменён  
- **Операция:** 1 Ожидает → 2 В работе → 3 Завершена / 4 Отменена  
- **План:** 1 Открыт, 2 Закрыт

## REST API (обзор)

Все маршруты под префиксом `/api/...`. Большинство требуют заголовок `Authorization: Bearer <token>`.

| Область | Контроллер / путь | Заметки |
|---------|-------------------|---------|
| Вход, профиль | `POST /api/auth/login`, `GET /api/auth/me` | login без JWT |
| Позиции плана | `GET /api/PlanPositions`, `GET .../{id}`, `PATCH .../{id}/status` | фильтры `guildId`, `month`, `year`; авто-фильтр по цеху для начальника/мастера |
| Статусы планов | `GET /api/PlanStatuses` | только чтение |
| МЛ | `GET /api/RoutingSheets`, `GET .../{id}`, `POST .../generate/{planPositionId}`, `PATCH .../{id}/status`, `POST .../{id}/split` | фильтры в т.ч. `guildId`, `partId`, даты создания; `generate` создаёт набор МЛ по деталям изделия; split — тело `{ "splitQuantity": n }` |
| Операции | `GET` (списки и по id), `PATCH .../assign-performer`, `DELETE .../performer`, `PATCH .../status`, `POST .../{id}/split` | ручное POST/PUT/DELETE операций сняты |
| Детали | `GET /api/Parts`, `GET .../{id}` | read-only |
| Состав изделия | `GET /api/ProductParts?productItemId=` | read-only |
| Изделия, статусы МЛ/операций | `GET /api/ProductItems`, `RoutingSheetStatuses`, `OperationStatuses` | read-only |
| Единицы, цеха, исполнители | `/api/Units`, `Guilds`, `Performers` | CRUD |
| Пользователи | `/api/Users` | `[Authorize(Roles = WorkshopChief)]` |

Полный интерактивный список — в **Swagger UI** после запуска API.

## База данных и миграции

- Контекст: `backend/Data/ApplicationDbContext.cs`  
- Миграции: `backend/Migrations/`  
- Команды (из каталога `backend`):

```bash
dotnet ef migrations add ИмяМиграции
dotnet ef database update
```

Пакет `Microsoft.EntityFrameworkCore.Design` уже подключён в `.csproj` для работы CLI. Команда `dotnet ef` идёт из глобального инструмента `dotnet-ef` (см. [Требования](#требования)).

Текущая схема включает миграцию `AddPartToRoutingSheets`, которая добавляет привязку маршрутного листа к детали и поддерживает генерацию отдельных МЛ по каждой детали изделия.

## Подсказки

- При смене пути к файлу БД обновите **`ConnectionStrings:DefaultConnection`** в `appsettings.json` или `appsettings.Development.json`.  
- Если измените HTTPS-порт API в **`backend/Properties/launchSettings.json`**, обновите **`frontend/vite.config.ts`** (прокси на `/api`) и переменную **`@baseUrl`** в **`backend/RoutingSheetsNew.http`**.

## Устранение неполадок

- **Ошибки SQLite / миграций** — удалите повреждённый `backend/Data/routing.db` (и при необходимости `-shm`/`-wal`) и перезапустите API; убедитесь, что каталог `backend/Data/` доступен на запись.  
- **CORS errors** — origin фронтенда должен совпадать с политикой в `Program.cs`.  
- **401 после входа** — проверьте время на машине (JWT lifetime), совпадение Issuer/Audience/Key.  
- **Прокси Vite (`ECONNREFUSED`)** — сначала запустите API (`dotnet run` в `backend/`). Убедитесь, что `vite.config.ts` указывает на тот же порт, что и профиль запуска: по умолчанию **HTTP 5145**, не 7050.
